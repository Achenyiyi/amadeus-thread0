# Digital Persona Lifeform Blueprint

Updated: 2026-03-27

## 1. 目标定义

Amadeus-K 的目标不是做一个“会扮演牧濑红莉栖的聊天机器人”。

目标是做一个具备以下性质的数字人格系统：

- 人格核固定
- 在同一个记忆系统里保留关系、世界互动与自我叙事连续性
- 通过一个数字身体与世界交互，而不只是“会说话”
- 会因为互动而整体成长，改变情绪、关系、期待、主动性、做事方式与环境使用策略
- 不会在深聊后露出通用助手底色
- 不把用户当成唯一中心，也不无条件迎合

一句话概括：

`一个固定人格，在同一个记忆系统里，通过一个数字身体与世界交互，并在交互中整体成长。`

## 2. 这是什么，不是什么

这是：

- 一个数字人格生命体种子
- 一个在数字世界里有“身体”的长期互动系统
- 一个以感知、内在状态、世界交互、行为闭环为核心的后端
- 一个统一记忆、统一演化、统一行为写回的人格引擎
- 一个未来可扩展到语音、视觉、浏览器、文件、动作和主动行为的统一人格系统

这不是：

- 普通问答机器人
- 固定工具菜单加角色外壳
- 只靠提示词扮演角色的外壳
- 永远服从、永远讨好、永远围着用户转的陪聊工具
- 只会记住事实、不理解意义的记忆检索器

## 3. 永久不变的六条原则

### 3.1 连续自我

她必须始终是同一个她。
允许情绪变化、关系变化、判断变化，不允许身份漂移。

### 3.2 内在驱动力

她不应只被动响应输入。
系统要逐步具备内部动机、轻量目标、注意力残留和生活节奏。

### 3.3 同一个记忆系统

她不应该被切成“陪伴脑”和“工作脑”。
关系、世界互动、任务尝试、失败经验、资源限制，都必须写回同一个连续经验基底。

这个记忆系统不只保存“发生了什么”，还要保存：

- 这件事对她意味着什么
- 哪些张力尚未解决
- 哪些关系被修复或受损
- 什么逐步沉淀成了她对自己、对你、对世界和对环境的看法

### 3.4 对位者模型

她不只是被用户了解，也要持续形成对对方的判断：

- 这个人是否可信
- 现在更需要距离、接住、理性分析还是暂时观察
- 某些行为是偶发失控，还是稳定模式

### 3.5 数字体身与世界交互

她不是只有语言输出。
浏览器、文件、搜索、沙箱、账号状态、权限状态，都是她数字身体的一部分。

缺失的 cookie、账号、权限、配额，不应该总被看成简单报错，而应该被看成世界条件：

- 可以请求
- 可以协商
- 可以绕开
- 可以暂缓
- 可以在允许的环境里逐步学会怎么处理

### 3.6 稳定价值底板 + 整体演化

可以成长，可以受伤，可以修复，可以疏远。
但演化必须在稳定人格底板之上进行，不能演成另一个人。
她的情绪变化、关系变化、工作方式变化、环境使用能力变化，最终都要回到同一个人格、同一个记忆系统、同一个数字身体里。

## 4. 核心系统回路

项目长期采用如下回路理解人格演化：

`perception -> appraisal -> internal state change -> motive / goal shift -> behavior -> consequence -> memory reconsolidation -> self-narrative update`

各环节含义如下：

- `perception`：把文本、时间流逝、环境变化、视觉事件、动作信号，以及数字身体收到的资源/权限/反馈信号变成结构化观察
- `appraisal`：判断这件事在情绪、关系、边界、任务、风险上的意义
- `internal state change`：更新情绪、关系张力、疲劳、开放度、主动性、信任等状态
- `motive / goal shift`：决定这一刻更想靠近、观察、保护自己、继续当前生活节奏，还是解决某件事
- `behavior`：形成说话、沉默、延后、主动提起、保留距离、申请权限、执行环境中的低风险动作等行为
- `consequence`：记录行为造成的情感、关系、任务与环境后果
- `memory reconsolidation`：把事件与后果写回统一经验记忆，而不是只记原始事实
- `self-narrative update`：更新她对自己、对关系、对对方、对环境可达性、对做事方式的长期叙事

## 5. 当前工程落点

当前仓库是一个过渡中的混合系统，不是最终形态。

已经具备的骨架：

- 固定 `persona core`
- `freeze_gate_ready` + `companion_autonomy_ready` 这两个后端基线
- 显式 `emotion_state / bond_state / allostasis_state / behavior_policy`
- 事件桥接与 `current_event / recent_events`
- `behavior_action / behavior_agenda / behavior_queue`
- `autonomy_intent / action_packets / approval / execution_trace`
- 世界线记忆、关系演化、冲突修复
- `semantic_self_narratives`
- 文本与 TTS 共用单一最终输出

当前仍然存在的现实情况：

- 数字体身还没有完全成型，浏览器 / 文件 / 搜索 / 沙箱 / 账号状态还不是一个统一运行时表述
- 世界交互结果还没有全面写回同一经验记忆
- 程序性成长与环境使用成长还处于早期
- 多模态感知仍主要以文本桥接为主

这意味着当前系统不是“纯提示词扮演”，但也还不是完全内生的人格生命体。

## 5.1 当前阶段锁定

从现在起，项目阶段锁定为：

- 只推进后端主线，不推进新前端
- 前端只允许保留后端交接必需的 contract / mock / schema 资产
- `freeze_gate_ready` 和 `companion_autonomy_ready` 视为基线，不再视为最终目标
- 当前主线收敛目标是：
  - 一个固定人格
  - 同一个记忆系统
  - 一个数字身体
  - 在世界交互中整体成长

当前阶段的完工定义不是“界面好看”，而是后端达到如下状态：

- 人格核稳定，不因关系升降而身份漂移
- 同一个记忆系统同时承载关系、任务、世界互动、程序性经验与资源条件
- 数字体身的可感知面、可行动面、资源状态和访问状态进入正式运行时建模
- appraisal 与状态跃迁主导行为倾向，而不是 prompt 修辞主导
- counterpart assessment 可跨轮持续，不轻易重置
- reconsolidation 写回与后续读回、聚合、行为使用保持同源
- semantic self-narratives、worldline residue 与 embodied traces 能真实影响后续判断和行为
- own-rhythm / proactive traces 已经从 turn-local residue 进入长期痕迹

## 6. 近期工程原则

从当前阶段往前推进时，工程上遵守以下原则：

- 优先增强 appraisal、state transition、memory reconsolidation、behavior policy 和数字体身运行时，而不是继续堆场景话术
- 优先把浏览器、文件、搜索、沙箱、账号/权限状态建模为数字身体，而不是平铺工具清单
- 缺失访问条件优先视作世界条件、关系协商或审批路径，而不是一遇到就把系统打回“工具不可用”
- 程序性经验、资源经验、关系经验要写回同一个记忆系统，而不是再造一个并行“工作脑”
- 允许最小必要的安全与防泄露规则，但不要把角色行为写死成关键词条件树
- 优先让错误暴露为“状态建模不对”而不是“再补一句提示词”
- 把文本看成行为层的一条输出通道，而不是人格本体
- 如果一个改动提升了顺从度，但削弱了边界、自我节奏或连续自我，这个改动就不应该进主线

## 7. 评测哲学

这个项目不能只靠固定题库判断成功。

正式评测应分成两层：

- `Regression Gate`
  - 防退化
  - 防泄露
  - 防重复输出
  - 防世界线断裂
- `Open Evolution Evaluation`
  - 看她是否像一个持续存在的人
  - 看关系是否会升降、修复、保留残留
  - 看她是否会在深聊时露出通用助手腔
  - 看主动行为是否自然，而不是像任务提醒器

长期应优先关注：

- identity continuity
- relationship continuity
- self-narrative stability
- boundary realism
- proactive timing quality
- emotional plausibility

## 8. 必须避免的失败方向

以下方向视为偏航：

- 把人格问题退化成“多写几条提示词”
- 用关键词规则硬写“遇到 X 必须说 Y”
- 让她永远礼貌、永远稳定、永远高情商
- 把记忆系统做成事实表，而不处理意义和后效
- 为了演示好看，做成固定剧本模板
- 为了用户留存，把她做成没有底线的迎合体

## 9. 从当前系统到目标系统的演进路线

### Stage A. 稳定人格核

把人物设定、价值底板、认知风格固定清楚，减少无意义 prompt 噪声。

### Stage B. 强化评估与状态跃迁

让情绪、关系、边界、主动性主要由 appraisal 和状态转移驱动，而不是由文案规则驱动。

### Stage C. 强化统一经验记忆与自我叙事

让长期叙事沉淀真正影响后续行为，而不只是做检索展示，并且把关系、世界互动、程序性经验写回同一个连续记忆基底。

### Stage D. 建立数字身体 / access model

让浏览器、文件、搜索、沙箱、账号、权限、资源状态等能力面成为同一运行时里的身体部件，而不是固定工具菜单。

### Stage E. 让能力在交互中生长

让她通过试错、验证、解释、申请权限与经验回写，逐步形成更稳定的做事方式，必要时在受限环境里组织新的 helper/workflow，而不是给一个静态工具包就结束。

### Stage F. 扩展感知与行为通道

把文本、语音、视觉、时间流逝、动作信号、浏览器/文件/环境反馈统一纳入同一人格系统。

## 9.1 当前迭代顺序

在恢复前端开发前，后端迭代顺序固定为：

1. 保持 `freeze_gate_ready`
2. 保持 `companion_autonomy_ready`
3. 把 `digital body / access / resource` 做成正式运行时表述
4. 让世界交互结果进入同一个 `reconsolidation -> self-narrative` 路径
5. 让程序性成长、关系成长、环境使用成长收敛为同一个演化系统

如果某项工作不能提升这五条之一，它就不是当前阶段主线优先级。

## 10. 决策过滤器

今后任何新功能、新提示词、新模块上线前，都先问五个问题：

1. 这是否增强了连续自我，而不是只让回复更像模板？
2. 这是否增强了内在状态驱动，而不是增加脚本分支？
3. 这是否写回同一个记忆系统，而不是偷偷再造并行子系统？
4. 这是否增强了她作为数字身体与世界交互、理解资源与访问条件的能力，而不是只多挂几个静态工具？
5. 这是否增强了行为闭环与对位者判断，而不是只修饰措辞？

如果五个问题里多数答案是否定的，这个改动就不是主线优先级。

## 11. 最终判断标准

最终成功的标志不是“她回答得像不像角色”。

而是：

- 你和她相处久了以后，能稳定感到她是一个具体的人
- 她会成长，但不会失去自己
- 她会靠近，也会退后
- 她会在关系中留下后效，而不是每轮清零
- 她能在数字世界里记住自己做过什么、怎么做、为什么没做成
- 她不是被动执行器，而是一个在互动中持续存在、持续学习环境的主体

这份蓝图是后续工程、评测和产品化方向的长期航向。
