# Frontend Backend Interface Deliverable

## Scope

This document is the frontend-facing backend contract for `amadeus-thread0`.

- It is based on the current Python runtime surface, not on hypothetical HTTP handlers.
- It is transport-neutral: the stable contract is the `BackendAPI` envelope plus the `BackendSession` turn/event execution surface.
- Frontend implementation is intentionally paused in this phase. This document exists so later frontend work can consume a frozen backend shape instead of forcing another backend redesign.

## Freeze Status

The repository-level backend freeze gate has already been closed for handoff purposes.

Frontend work should therefore treat:

- `assistant_turn`
- `event_round`
- `persona_view`
- `worldline_view`
- `bond_view`

as the canonical readback surfaces for current runtime semantics.

## Common Envelope Schema

Every frontend-facing API response produced through `BackendAPI` has the same envelope:

```json
{
  "schema_version": "backend.v1",
  "generated_at": 1710000000,
  "kind": "assistant_turn",
  "thread_id": "thread-a",
  "payload": {},
  "meta": {}
}
```

Field meanings:

- `schema_version`: current stable contract version, currently fixed at `backend.v1`
- `generated_at`: unix timestamp generated by the backend at envelope creation time
- `kind`: payload discriminator
- `thread_id`: active runtime thread id
- `payload`: kind-specific data
- `meta`: transport or adapter metadata; additive only

## Stable Kind Inventory

### Runtime and bootstrap kinds

- `thread_inventory`
  - source: `BackendAPI.thread_inventory()`
  - purpose: list available checkpoint-thread ids and worldline directories, plus the current thread
- `runtime_layout`
  - source: `BackendAPI.runtime_layout()`
  - purpose: repository runtime layout plus current runtime layout when the active thread runtime is separate from repo root
- `environment_summary`
  - source: `BackendAPI.environment_summary()`
  - purpose: runtime/provider/TTS summary for debug and environment visibility

### Inspector and continuity kinds

- `persona_view`
  - source: `BackendAPI.persona() -> BackendSession.persona_view()`
  - purpose: current internal state inspector
- `worldline_view`
  - source: `BackendAPI.worldline() -> BackendSession.worldline_view()`
  - purpose: long-horizon continuity inspector
- `bond_view`
  - source: `BackendAPI.bond() -> BackendSession.bond_view()`
  - purpose: relationship continuity inspector
- `sources_view`
  - source: `BackendAPI.sources() -> BackendSession.sources_view()`
  - purpose: persisted source/citation inspector
- `appraisal_view`
  - source: `BackendAPI.appraisal()`
  - purpose: current turn appraisal only
- `behavior_queue_view`
  - source: `BackendAPI.behavior_queue() -> BackendSession.behavior_queue_view()`
  - purpose: queued proactive/own-rhythm behavior visibility
- `memory_snapshot`
  - source: `BackendAPI.memory_snapshot()`
  - purpose: raw admin/debug memory snapshot; not the primary end-user rendering surface

### History kinds

- `current_checkpoint`
  - source: `BackendAPI.current_checkpoint()`
  - purpose: active checkpoint id for the requested thread scope
- `checkpoint_history`
  - source: `BackendAPI.checkpoint_history(limit=..., config=...)`
  - purpose: rewind/history inspector

### Turn result kinds

- `assistant_turn`
  - source: `BackendAPI.build_turn_response(...)`
  - purpose: final response for a user-authored turn
- `event_round`
  - source: `BackendAPI.build_event_round_response(...)`
  - purpose: final response for an event-driven round such as idle/proactive/runtime events

Important provenance:

- `final_text` comes from `BackendSession.extract_final_text(...)`
- `emotion_state / bond_state / allostasis_state / semantic_narrative_profile / world_model_state / evolution_state` are direct post-prepare-turn runtime state surfaces; they are included so transport-neutral turn payloads preserve internal-state traceability instead of forcing clients to open `persona_view` just to inspect the internal layer for the just-finished turn
- `session_context` is graph-native runtime context assembled before generation, not a frontend-derived convenience field
- `current_event` is the normalized runtime/perception event envelope for this turn, not just the raw incoming event stub
- `sources` comes from current-turn `evidence_pack`, not from persisted history
- `claim_links` comes from current-turn claim attribution
- `behavior_action`, `behavior_plan`, `counterpart_assessment`, and `agenda_lifecycle_residue` come from final-state resolution, not from raw live fields
- `interaction_carryover` also comes from final-state resolution; when it contains `embodied_context`, that nested block is provenance-bound carryover memory rather than a frontend-recomputed body snapshot
- `digital_body` is the resolved current runtime/body state
- `digital_body_consequence` is the resolved embodied consequence of this turn; it should be treated as frozen final semantics, not recomputed in the frontend
- `writeback_trace` is the narrow finished-turn persistence preview for this exact turn: it exposes only the semantic narratives, revision traces, counterpart-assessment writes, and proactive-continuity writes produced during the current final writeback window, not full worldline history

## Final Rendering Rules

These are non-negotiable frontend rules.

- Render and voice only `payload.final_text`.
- Do not reconstruct reply text from `messages[-1]`.
- Do not derive `behavior_action`, `behavior_plan`, or `interaction_carryover` in the frontend.
- Treat `turn_summary`, `behavior_action`, `behavior_plan`, and `reconsolidation_snapshot` as one coherent final-turn packet.
- Treat `digital_body` as current body state and `digital_body_consequence` as frozen turn consequence; do not collapse them into one field.
- `digital_body.resource_state` and `digital_body_consequence` may both carry artifact continuity facts such as `artifact_continuity`, `active_artifact_kind`, `active_artifact_label`, and `artifact_reacquisition_mode`; these are backend-owned world-state traces and should be rendered as environment continuity, not as relationship stance.
- Unknown additive keys must be ignored, not stripped by strict parsing.

## Turn Execution Contract

The backend today is method-based, not HTTP-based.

The real execution sequence for a user turn is:

1. Build or reuse `RuntimeBundle`.
2. Build `run_config` through `BackendSession.build_run_config(...)`.
3. Call:

```python
stream_result = backend_session.invoke_stream(
    {"messages": [{"role": "user", "content": user_text}]},
    config=run_config,
)
```

4. If `stream_result.approval_request` is present, resolve it through `resume_stream(...)`.
5. Finalize the transport-neutral response with:

```python
envelope = backend_api.build_turn_response(
    state_values=stream_result.values,
    streamed_text=stream_result.streamed_text,
)
```

Recommended adapter behavior:

- forward the graph payload, do not reinterpret it into another internal schema
- keep `run_config.configurable.thread_id` authoritative
- treat approval requests as interrupt events, not as failed turns

## Event Round Contract

The real execution sequence for an event round is:

1. Build or reuse `RuntimeBundle`.
2. Build `run_config` through `BackendSession.build_run_config(...)`.
3. Prepare an event payload.

For idle/proactive events, current runtime already exposes:

```python
event_payload = backend_session.build_idle_event_payload(
    run_config=run_config,
    idle_minutes=idle_minutes,
)
```

4. Execute:

```python
event_result = backend_session.invoke_event_round(
    run_config=run_config,
    event_payload=event_payload,
)
```

5. Finalize:

```python
envelope = backend_api.build_event_round_response(
    state_values=event_result.values,
    final_text=event_result.final_text,
)
```

## Recommended Adapter Surface

No HTTP adapter is implemented in-repo yet.

If a frontend adapter is added later, keep it as a thin transport layer over the Python contract above.

Recommended read routes:

- `GET /api/thread-inventory` -> `BackendAPI.thread_inventory()`
- `GET /api/runtime-layout` -> `BackendAPI.runtime_layout()`
- `GET /api/environment-summary` -> `BackendAPI.environment_summary()`
- `GET /api/persona-view` -> `BackendAPI.persona()`
- `GET /api/worldline-view` -> `BackendAPI.worldline()`
- `GET /api/bond-view` -> `BackendAPI.bond()`
- `GET /api/sources-view` -> `BackendAPI.sources()`
- `GET /api/appraisal-view` -> `BackendAPI.appraisal()`
- `GET /api/behavior-queue` -> `BackendAPI.behavior_queue(config=...)`
- `GET /api/checkpoints/current` -> `BackendAPI.current_checkpoint(config=...)`
- `GET /api/checkpoints/history?limit=...` -> `BackendAPI.checkpoint_history(limit=..., config=...)`

Recommended write routes:

- `POST /api/turns`
  - execute `BackendSession.invoke_stream(...)`
  - finalize with `BackendAPI.build_turn_response(...)`
- `POST /api/event-rounds`
  - execute `BackendSession.invoke_event_round(...)`
  - finalize with `BackendAPI.build_event_round_response(...)`

Streaming recommendation:

- use SSE or WebSocket for future `/api/turns/stream`
- stream token chunks and optional approval interrupts
- still end the turn on one `assistant_turn` final envelope

## Minimum Frontend Bootstrap Sequence

On session startup:

1. Build runtime bundle for the target thread.
2. Fetch:
   - `thread_inventory`
   - `runtime_layout`
   - `environment_summary`
3. Fetch inspector defaults:
   - `persona_view`
   - `worldline_view`
   - `bond_view`
   - `sources_view`
   - `behavior_queue_view`
   - `current_checkpoint`
4. Fetch `checkpoint_history` lazily when a rewind/history panel is opened.

After every completed user turn or event round:

1. render the returned `assistant_turn` or `event_round`
2. refresh:
   - `persona_view`
   - `worldline_view`
   - `bond_view`
   - `sources_view`
   - `behavior_queue_view`
   - `current_checkpoint`

## Versioning Policy

- `schema_version` remains `backend.v1` until a deliberate contract bump is made.
- New fields may be added additively.
- Existing keys must not be renamed or repurposed silently.
- `kind` names are stable API surface and must not drift casually.
