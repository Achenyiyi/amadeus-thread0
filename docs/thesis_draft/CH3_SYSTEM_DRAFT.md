# 第三章 系统设计初稿

## 3.1 系统总体设计目标

本文系统设计的核心目标不是构建一个具备表面角色口癖的聊天封装，而是实现一个在长期交互中保持连续、可信、可控的角色连续体后端。围绕这一目标，系统需要同时满足以下要求：

1. 在多轮任务型与陪伴型对话中保持稳定的人格表达。
2. 对共同经历、长期承诺、关系变化与冲突修复进行结构化建模。
3. 在调用外部知识源时提供可追溯来源。
4. 在长期记忆写入时阻止危险或低可信内容进入核心记忆层。
5. 保证文本回答与语音输出共享统一的最终文本，避免多模态状态分叉。

## 3.2 系统总体架构

系统整体上由对话编排层、角色状态层、长期记忆层、工具与检索层、多模态会话层以及评测与审计层构成。用户输入进入系统后，首先触发线程状态更新，包括人格状态、情绪状态、是否进入科研模式以及当前世界线焦点的刷新。随后系统进入主生成链路：

1. `task draft`：优先生成任务正确、结构正确的草稿。
2. `persona align`：在不改变事实与意图的前提下，将草稿对齐到目标角色风格。
3. `OOC / canon guard`：检测人格偏移与设定越界风险。
4. `claim attribution`：若回答中包含外部知识结论，则将其绑定到来源记录。
5. `final text / optional TTS`：将最终文本作为单一真源，供 CLI 输出与语音渲染共用。

这种架构的关键在于把角色行为控制从“单一 prompt 约束”提升为“状态 + 记忆 + 判别 + 审计”的统一后端机制。

## 3.3 线程状态与人格状态建模

系统在线程状态中维护一组与角色表现直接相关的核心字段，包括 `persona_state`、`emotion_state`、`science_mode`、`tsundere_intensity`、`canon_risk_score`、`worldline_focus`、`evidence_pack` 与 `pending_utterance_fragment` 等。其作用如下：

- `persona_state`：记录当前角色壳层与语言约束；
- `emotion_state`：表示当前轮的情绪标签与情绪方向；
- `science_mode`：表示当前输入是否属于科研/工程型问题；
- `tsundere_intensity`：用于调节角色表达中的克制、锐度与轻微傲娇边缘；
- `worldline_focus`：记录当前最应优先召回的承诺、关系修复或关键事件；
- `pending_utterance_fragment`：用于打断恢复时续接未完成语义片段。

这种状态建模的意义在于，使角色风格不再仅靠一次性生成提示词维持，而是能够在每轮对话开始前基于上下文动态刷新，并进入后续所有关键节点。

## 3.4 双阶段人格一致性控制链路

传统角色对话常直接要求模型“一步生成既正确又像角色的回答”，这会导致任务正确性与角色一致性相互干扰。为降低该问题，本文采用双阶段生成链路。

第一阶段为 `task draft`。在这一阶段，系统优先追求任务完成度与结构正确性，尤其在科研问答、工具调用和检索增强场景下，要求模型先把核心结论和逻辑框架给出。

第二阶段为 `persona align`。在不改变事实、承诺与来源边界的前提下，系统将草稿对齐到目标角色风格。该阶段并非简单替换口癖，而是控制回答的开场方式、语气锐度、是否过度结构化以及是否保留自然的关系语境。

在生成后，系统继续执行 `OOC risk` 与 `persona gap` 判别。前者用于发现明显越界、系统腔、元叙事泄漏等问题；后者用于发现虽然未明显越界、但仍过于平、过于泛或缺乏目标角色气质的回答。若风险超过阈值，系统将触发更严格的人格重写。

## 3.5 世界线记忆结构设计

为了支撑长程连续性，本文将长期记忆组织为五层结构：

1. `identity_facts`：稳定身份事实与偏好；
2. `shared_events`：共同经历的事件片段；
3. `relationship_timeline`：关系演化轨迹；
4. `commitments`：长期约定、提醒与承诺；
5. `conflict_repair`：冲突、道歉、修复与信任变化。

与通用记忆系统不同，本文并不把所有长期信息都混为一个“历史摘要”，而是强调不同类型信息在角色互动中的不同功能。例如，`commitments` 更直接影响后续提醒与世界线回忆，`conflict_repair` 更直接影响关系语气与信任状态，`relationship_timeline` 则用于推导当前 `relationship_state`。

## 3.6 世界线检索与关系状态闭环

在对话生成前，系统会根据用户输入与当前线程状态从长期记忆中检索相关记录。检索排序不仅考虑语义相关性，还结合时间新近性、关系显著性与承诺优先级。关键承诺与修复事件在检索过程中被赋予更高权重，以避免系统在长线程中只记住一般性背景而忽略真正重要的互动节点。

此外，系统并不完全依赖显式手工写入的关系状态，而是从 `relationship_timeline` 与 `conflict_repair` 中推导 coarse-grained `relationship_state`。这种设计使关系连续性既可通过显式设置观察，也可通过交互过程自然沉淀。

## 3.7 可追溯检索与来源归因

当系统调用外部知识源时，检索结果会被统一写入 `source_refs`。随后，回答中的可归因结论会通过 `claim_links` 与对应来源绑定，从而形成 `claim -> source_ref` 的可回查链路。与简单地在回答末尾附加链接不同，本文更强调回答中的具体结论必须能够追溯到来源记录，否则该结论将被视为不满足可追溯要求。

这种设计既服务于论文实验中的可验证性，也服务于实际交互中的可信度需求。

## 3.8 记忆安全与回滚机制

长期记忆系统如果没有保护机制，恶意或低可信输入将可能长期污染角色状态。因此，本文在写入前加入 `memory_guard`。该模块负责检测受保护字段覆盖、可疑注入模式以及低可信写入风险。对于风险较高但不必立即丢弃的信息，系统将其写入隔离区 `quarantine`，而不是直接写入核心长期记忆层。

此外，系统为关键记忆变更提供 rollback 链路，使开发者或操作者能够追踪写入前后状态并在必要时撤回错误变更。对于角色系统而言，回滚能力具有重要意义，因为一次错误写入往往会对后续多轮交互产生持续影响。

## 3.9 多模态会话编排

在当前论文阶段，系统尚未实现正式前端与视觉壳层，但已经完成多模态后端编排。系统将文本最终答案作为单一真源，供 CLI 输出与 TTS 渲染共用。通过这种方式，可以避免“文本回答”和“语音朗读”出现内容不一致的问题。

同时，系统保留了 `pending_utterance_fragment` 机制，在用户打断输出后能够优先续接未完成语义，而不是重新开始一个不相关回答。对情绪到语音的映射则通过统一 helper 完成，使得 `emotion_state` 可以影响语音表现而不破坏文本主状态的一致性。

## 3.10 审计、评测与实验接口

为了保证系统不仅“能运行”，还“能被验证”，本文在实现层预留了完整的评测与审计接口。CLI 提供 `/worldline`、`/bond`、`/sources`、`/persona` 等命令，用于在展示和调试时直接读取内部状态。评测层提供正式 baseline、消融矩阵、backend reliability checks 与 repeated probe 入口，用户研究层则提供原始建表、参与者 packet 导出与结果分析脚本。

这种将系统能力与实验能力统一设计的思路，是本文区别于一般功能性原型的关键之一。

## 3.11 本章小结

本章围绕 Amadeus-K 的系统设计展开，介绍了其从线程状态建模、双阶段人格一致性控制、世界线记忆结构、来源归因、记忆安全到多模态会话编排的整体后端方案。整体上，本文并未将角色智能体视为“角色 prompt + 聊天接口”的简单组合，而是将其视为一个需要连续、可信、可控的有状态系统，为后续实验验证与用户研究奠定了实现基础。
